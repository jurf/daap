7.3 Retry

Do not fail because of transient errors

This pattern is based on the Retry pattern by [83], Retry with backoff by [84], Request-Response with Retry by [85], At-Least-Once delivery by Fehling et al. [15, p. 144, 86] and Request/Reaction by [13, p. 114].

7.3.1 Context

A service is connecting to an external resource and this request is either idempotent or there is some sort of duplicate handling in place. A failure to get a response would negatively affect the currently running operation and getting a response, even with added latency, would be beneficial. The response either signalled the failure is transient explicitly (i.e. with a busy signal [14, p. 83]) or there is a probability it is.

7.3.2 Solution

When a failure occurs, retry the request. However, there are several option on when to retry and for how long.

If latency is an issue, e.g. an interactive situation, and this type of failure is known to be transient or to resolve quickly, retry immediately, with a fixed delay or with a linearly increasing delay, depending on the balance needed.

If this is a long-running operation, an exponentially increasing delay (also known as exponential backoff [87]) would be more suitable to avoid collisions or overloading the service. [14, p. 88, 83]

To prevent collisions, consider adding jitter to the delay [88].

The service might also provide an estimate of how long to wait, e.g. if it employs a Rate-Limitter (see § 7.5).

Retry until the failure is recognised as non-transient, or a timeout or a maximum number of retries is reached (see figure 7).

7.3.3 Potential issues

If a service is failing due to overload, retrying the request might exacerbate the problem. If the retries are long-running, employing an inappropriately-designed backoff algorithm might lead to collisions [88] (also known as the thundering herd problem [89]).

Depending on the implementation, this pattern might introduce delays or increased load (especially if the problem is mistakenly categorised as transient). If the service is consumed by a client, it might be better to let them react to the failure on their own instead [11, p. 113]. It may also result in duplicate requests if the issues were caused by the network and the retry logic was not implemented correctly.

7.3.5 Related patterns

  • Use a Circuit Breaker for avoiding retries if a failure is long-lasting (see § 8.1)

  • Queue-Based Load Leveling typically implements retry logic (see § 7.2)

  • Retries can be implemented in a reusable Ambassador (see § 5.2)
