4.3 Asynchronous Request–Reply

Asynchronous two-way communication

This pattern is based on the Request–Reply pattern defined by Hohpe; Woolf [4, p. 147, 50], the Asynchronous Request–Reply pattern by [51], the Asynchronous Request–Response pattern by [52], the Request-Response Communication pattern by [3, p. 104], and partly on the Messaging pattern by Richardson [20, p. 87, 47] and Decoupling Middleware pattern by [11, p. 141].

The patterns have varying implementations, but all share the same structure and purpose. An overview is given below.

4.3.1 Context

Synchronous messaging has become a bottleneck, but using only Publisher–Subscriber (see § 4.2) is not an option, as either

  1. The client expects a response

  2. One of the communicating parties cannot use a messaging broker due to technical limitations (e.g. a web browser), or when long-running connections are not feasible

4.3.2 Solution

There are three potential ways to implement this pattern.

  1. Use a message broker (see § 4.2) and configure two topics: one for the request and one for the response. The sender sends a message to the request topic and waits for a response on the response topic.

  2. Use synchronous communication to send the request, and then use polling to check for a response on a pre-determined endpoint. This option is less efficient as there will be more communication overall, but might be easier to implement with existing infrastructure or tools. Efficiency can be improve by the receiver replying with an estimate on the request completion [51].

  3. Use synchronous communicationc and a callback mechanism. The sender sends a request and a callback address. The receiver processes the request and sends the response to the callback address. [52]

In both circumstances, attaching a unique identifier to the request is desirable in order to match the response to the initial request.

4.3.3 Potential issues

When compared to synchronous communication, this pattern brings increased latency and complexity. If implemented using synchronous calls, Retry logic (see § 7.3) might need to be employed to handle transient failures.

Using synchronous communication is faster, simpler and more efficient, if the requirements allow for it.

Using this pattern to communicate with multiple parties may result in requests being processed out of order. This can be mitigated by using a Correlation Identifier [4, p. 61, 53].

4.3.4 Example

4.3.5 Related patterns

  • While not necessary, Queue-Based Load Leveling can be used to improve handling peaks in demand (see § 7.2). If doing so, this pattern is, in essence, two Publisher–Subscriber applications, the other being used in reverse (see § 4.2).

  • Retry logic might be necessary if using synchronous communication (see § 7.3)
