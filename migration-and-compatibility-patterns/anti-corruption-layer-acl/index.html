Distributed Application Architecture Patterns

11.1 Anti-Corruption Layer (ACL)

Mediate between modern and legacy systems

This pattern is based on the Anticorruption Layer1 pattern defined by [155, p. 364], later by Richardson [20, p. 446, 156], [157] and [158].

It has has a similar notion to the Adapter2 pattern by Burns [55, later 22, p. 31], although the latter focuses predominantly on its usage in regards to enabling Health Monitoring (see § 7.4) and is deployed as a Sidecar (see § 5.1).

Fowler’s Gateway [7, p. 466] can be seen as an implementation of this pattern [159, 160].

11.1.1 Context

An application needs to work with an external system that satisfies one of the following conditions.

  1. It uses outdated technologies that would bring in unwanted dependencies or are incompatible

  2. It has a poorly designed (corrupted) API

  3. It is impractical to change (e.g. a legacy system)

  4. It uses different model semantics

11.1.2 Solution

Implement an anti-corruption layer that isolates the two systems, acting as a intermediary that translates between them.

This layer can either be a separate software layer or a different service (see figure ¿fig:acl?).

11.1.3 Example

A video service needs to fetch data about YouTube videos into its database. However, YouTube uses a complex data structure that adds needless complexity, as the service only needs a small subset of the data.

An anti-corruption layer can be used to convert between the different domain models, shielding the service from any future API changes and separating the concerns (see figure 16) [159].

11.1.4 Potential issues

Any data transformation inevitably causes increased increased latency. When scaling the system up, the ACL may become a bottleneck or a single point of failure. If the underlying service is scalable, the ACL has to be designed with that in mind.

If the ACL implemented as an additional service, it adds maintenance cost. Furthermore, if it is employed as a short-term solution, the ACL needs to be tracked as technical debt.

Lastly, there is a risk of persisting corruption – if the API of the ACL is badly designed, the “corruption” will be propagated anyway.

11.1.5 Related patterns

  • The ACL can be deployed as a Sidecar for increased performance (see § 5.1)

  • The Incremental Replacement (Strangler Fig) pattern can be used to phase out the underlying system and ACL gradually (see § 11.2)

  • A Messaging Bridge can be used if systems use incompatible messaging systems (see § 11.3)

11.1.6 Further Reading

  • If there is no underlying issue with the system you want to adapt, the Facade [24, p. 175, 161], Adapter [24, p. 139, 162] and Mediator [24, p. 273, 163] patterns may still be applicable
  • Fowler’s example on how to use the anti-corruption layer to refactor code working with external services [159]
