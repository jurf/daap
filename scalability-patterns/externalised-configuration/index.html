6.4 Externalised Configuration

Centralised configuration management

To end the chapter on scalability, it is important to note that not all problems caused by an application scaling up need to have complicated solutions as they might not be a bottleneck. One such example is configuration management.

This pattern has several different names; we chose Chris Richardson’s version [20, p. 361] as we felt it best represented the core idea.

6.4.1 Context

When creating many deployments in a system, it might become challenging to properly configure or update the configuration of each instance. This causes the system to become inflexible and increases probability of user error, which can lead to failures or security issues.

6.4.2 Solution

Centrally manage the configurations of each service. The configuration is either requested by the services themselves when they startup, or it is requested by the infrastructure when creating each service.

6.4.3 Potential issues

The configuration might become a single point of failure. If the services themselves request the configuration, changing the configuration store implementation or location might require changes in all services.

If the number of services is very high, the services change frequently, or the configuration needs to be update online, the configuration store could potentially become a bottleneck.

6.4.5 Related patterns

The Identity Provider uses the same concept, but is used for accommodating security rather than scalability (see § 9.1).

6.4.6 Further reading

    • Pull-based externalized configuration described in this section [20, p. 363]

    • Push-based externalized configuration for configuring while creating a service instance [20, p. 362]

  • Gregor Hohpe and Bobby Woolf’s Control Bus pattern in Enterprise Integration Patterns is a similar concept, but rather using messaging to control other services [4, p. 477, 74]

  • The Managed Configuration pattern in Cloud Computing Patterns [15, p. 247, 75]

  • Azure Architecture Center’s External Configuration Store pattern [76]; also see the very similar Edge Workload Configuration pattern focusing more on IoT use cases [77]
