6.2 Partitioning (Sharding)

Scale out by separating tasks or data into logical partitions

In a certain way Competing Consumers (§ 6.1) already had some sort of partitioning in it, as it partitioned the tasks in time. This partitioning was not predetermined, but this is an idea we can expand further.

This pattern is sometimes known as sharding [3, p. 429], and has a host of subtypes in database design, but here we will focus on the universal concept. One might also argue that partitioning tasks and data is a very different concept, and they are sometimes separated [67, 68], but in fact they result in the exact same architecture, so similarly to § 4.3 we decided to embrace the duality and keep them together.

[22, p. 59]

6.2.1 Context

The data store or a specific task has become the bottleneck, or they surpass the capacity of a single machine, but it is possible to uniformly distribute them using some kind partitioning logic.

6.2.2 Solution

Instead of creating one shared data store or service, create multiple instances and divide the data or tasks between them (see figure 6.2).

Since each partition is managed by exactly one instance, this mean the order of operations is guaranteed within a partition, in contrast to Competing Consumers (see § 6.1).

If partitioning data, choose a fixed size of partitions to avoid having to repartition the data if the number of data stores changes [5, p. 219].

6.2.3 Potential issues

Data partitioning logic design can be complex and it is fundamental for the performance of the system. Choosing an incorrect scheme might require expensive repartitioning. [3, p. 429]

Since one partition is managed by exactly one instance, additional resiliency measures are needed to handle failures.

6.2.4 Related patterns

  • Even though the partitions do not increase the overall resiliency of the system per se, this can still be used to ones advantage to prevent cascading failures. Introducing logical partitions for this purpose is known as the Bulkhead pattern (see § 7.1)

  • An Ambassador can used to implement sharding logic on the client side (see § 5.2) [22, p. 22]

  • The router can implement Cache-aside to improve performance (see § ¿sec:cache-aside?)

6.2.5 Further reading

Partitioning tasks usually goes by different names, usually as a solution to the ordering issue of Competing Consumers.

  • Chris Richardson on partitioning messaging channels in Microservice Patterns [20, p. 94]

  • The Sequential Convoy pattern in the Azure Architecture Center [68]

Partitioning data is typically discussed in the context of databases as sharding.

  • See Unmesh Joshi’s Fixed Partitions pattern if the amount of participating data stores can be variable [5, p. 219] and Key-Range partitions to optimise for queries with ranges [5, p. 243]

  • Sam Newman on data partitioning in Building Microservices [3, p. 426]

  • Bill Wilder’s Database Sharding pattern in Cloud Architecture Patterns [14, p. 67]

  • Brendan Burns’ Sharded Services pattern in Designing Distributed Systems [22, p. 59]

  • The Sharding pattern in the Azure Architecture Center [67]

  • Shard (database architecture) on Wikipedia. Also see Data striping for a similar concept in storage systems

6.2.6 Deployment Stamps

An interesting sub-case of partitioning is partitioning users. If the use cases of the system allow for creating partitions of users that do not have to share data, creating a distributed system might not be necessary at all (see figure 6.2.6).

Partitions corresponding to geographic regions inherently facilitate low latency (until the users travel).
